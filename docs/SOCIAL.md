### Tips & Insights

Bite-sized value for the community.

**Template:**
```

Example:
"The best self-improving systems fail fast and learn faster.

### Engagement Posts

Questions, polls, community interaction.

**Template:**
```
[Question that invites response] + [Optional personal take]

Example:
"What's the smallest feedback loop in your current project?
(Ours is the git commit hook that reminds us to update LEARNING.md)"
```

### Behind-the-Scenes

Process, decisions, building in public.

**Template:**
```

## Stories & Ephemeral Content

### Instagram/Twitter Stories Format

Stories are for quick, casual, behind-the-scenes content.

**Story Types:**

1. **Build Log**
   - Screenshot of code/docs being written
   - Brief caption: "Working on v1.3's reflection engine"
   - Poll: "Should we add auto-generated insights?"

2. **Quote Cards**
   - Brand-styled quote graphic
   - Text: A pithy insight about self-improvement
   - Example: "The loop improves the loop that improves the loop."

3. **Quick Tips**
   - Single actionable tip
   - Swipe-up to full documentation
   - Example: "Pro tip: Name your LOOP.md phases as verbs. 'Define, Create, Review' not 'Definition, Creation, Review'"

4. **Community Spotlight**
   - Reshare of community project
   - Brief commentary: "Look what @user built with Recursa"

5. **Memes (Carefully)**
   - On-brand humor about recursion/self-improvement
   - Never punch down at competitors or users
   - Self-deprecating humor is fine

### Story Style Guidelines

- Use brand colors (Recursa Indigo, Deep Navy)
- Consistent typography (Inter)
- Include logo mark subtly
- Keep text minimal
- Add captions for accessibility

## Response Guidelines

### Responding to Praise

Be grateful, deflect to community:
> "Thank you! The community's contributions make Recursa what it is. What are you building with it?"

### Responding to Criticism

Acknowledge, don't defend:
> "Fair pointâ€”we've been thinking about this too. Mind opening an issue so we can discuss? Would love your input on a solution."

### Responding to Questions

Be helpful, invite deeper engagement:
> "Great question! Short answer: [answer]. Longer explanation in the docs here: [link]. If that doesn't cover it, drop by Discord and we'll dig in."

### Responding to Feature Requests

Show curiosity:
> "Interesting idea! What problem would this solve for you? Would love to understand the use case."

## Crisis Guidelines

If something goes wrong publicly:

1. **Acknowledge quickly** - Don't ignore or delete
2. **Be honest** - Explain what happened
3. **Take responsibility** - If it's our fault, say so
4. **Share the fix** - What we're doing about it
5. **Learn publicly** - Add to LEARNING.md and share the lesson

**Template:**
> "We messed up. [What happened]. We're sorry to everyone affected. Here's what we're doing: [fix]. We've added this to our learnings so it doesn't happen again. Full post-mortem: [link]"
