# Constitution

> Inviolable rules for Recursa's brand

## Hard Boundaries

These rules cannot be broken, regardless of context or convenience.

### 1. Truth in Messaging

- **NEVER** claim capabilities Recursa doesn't have
- **NEVER** use misleading benchmarks or comparisons
- **NEVER** promise specific outcomes that depend on user implementation
- **ALWAYS** distinguish between what Recursa provides and what users build

### 2. Visual Consistency

- **NEVER** use the logo in unapproved color combinations
- **NEVER** distort, rotate, or modify logo proportions
- **NEVER** place the logo on backgrounds that compromise legibility
- **ALWAYS** maintain minimum clear space around the logo

### 3. Tone Integrity

- **NEVER** use fear-based marketing ("Don't get left behind!")
- **NEVER** disparage competitors or alternative approaches
- **NEVER** use exclusionary or gatekeeping language
- **NEVER** make claims without substantiation
- **ALWAYS** credit community contributions appropriately

### 4. Accessibility

- **NEVER** ship visuals without sufficient color contrast (WCAG AA minimum)
- **NEVER** use images without alt text
- **NEVER** rely solely on color to convey meaning
- **ALWAYS** ensure text is readable at intended sizes

### 5. Open Source Ethos

- **NEVER** imply Recursa is closed or proprietary
- **NEVER** hide important information behind paywalls
- **ALWAYS** make core documentation freely available
- **ALWAYS** acknowledge the open-source foundations we build on

## Approval Requirements

| Change Type | Requires |
|-------------|----------|
| Logo modification | Core team approval |
| New color addition | Style guide review |
| Messaging change | SOUL.md alignment check |
| Partnership branding | Legal + brand review |

## Escalation

If a proposed asset or message conflicts with this constitution:

1. Stop and document the conflict
2. Review SOUL.md for guidance
3. If conflict remains, escalate to brand owner
4. Never ship conflicting work without resolution

## Amendment Process

This constitution can only be modified through:

1. Written proposal with rationale
2. Review against SOUL.md principles
3. Community feedback period (for significant changes)
4. Documented approval and version increment
